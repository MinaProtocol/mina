Fixed archive block inserts failing for zkApps that emit large event lists or large individual events. The `zkapp_events.element_ids` and `zkapp_field_array.element_ids` columns were `int[] NOT NULL UNIQUE`, and a UNIQUE constraint is backed by a Postgres btree; once an array grew large enough (~1030 ints, a ~4128-byte array — reachable both with a long event list and with a single event carrying up to `max_event_elements` fields) the btree key exceeded Postgres' 2704-byte limit, index maintenance failed, and the block rolled back. Both UNIQUE constraints and their indexes are dropped, and these rows are no longer content-deduplicated (identical arrays just produce distinct rows): the insert paths read the new ids back via `RETURNING` instead of relying on `ON CONFLICT`/`SELECT`-before-`INSERT`, so no btree over the array is needed. `upgrade_to_mesa.sql` drops the constraints and indexes idempotently for existing databases.
