Emit an `account_creation_fee_via_zkapp` Rosetta operation for an account created by a zkApp account update, closing a `check:data` reconciliation gap on the zkApp path (the user-command and internal-command paths already emitted their equivalents). The fee is charged to the created account **only** when that account update carries `implicit_account_creation_fee`, i.e. when the ledger paid the fee by shrinking the update's own `balance_change` so the resulting balance is `balance_change - fee`; the amount is read from `accounts_created.creation_fee`, so it stays correct across hardforks. When the flag is unset, the creating command funds the fee out of *other* account updates' negative balance changes — already emitted as `zkapp_balance_update` operations — and the created account's balance is exactly its `balance_change`, so no fee operation is emitted; doing so would double-count the fee and, whenever the created account's `balance_change` is zero, drive its Rosetta balance negative and abort `check:data`. Applies to both `/block` and `/search`. Fixes #19044.
